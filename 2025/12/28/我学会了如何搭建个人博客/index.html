我学会了如何搭建个人博客

发表于2025-12-28|更新于2025-12-29|学习笔记
|浏览量:

就在刚刚，我成功搭建了属于自己的个人博客，使用的是 Hexo + Butterfly 主题。下面记录的是从零开始搭建博客的全过程，以及在这个过程中遇到的各种问题与解决思路。

一、准备工作

我选择在本地 D 盘创建博客项目文件夹，路径如下：

D:\BlogProjects

需要注意的是，在 Windows 系统下，如果命令行默认位于 C 盘，必须先切换盘符，否则后续命令无法在目标目录下执行。正确的操作顺序是：

D:
cd D:\BlogProjects

在安装 Hexo 之前，需要确保本机已经安装 Node.js 和 npm，并且版本可正常使用。

二、初始化 Hexo

在一个空目录中执行以下命令：

hexo init .
npm install

Hexo 会自动生成基础目录结构，包括：
_config.yml
source
themes
package.json

如果当前目录不为空，hexo init 会直接报错，因此需要提前清空文件夹，或将原有内容备份。

三、安装 Butterfly 主题

进入 themes 目录：

cd themes

克隆 Butterfly 主题仓库：

git clone https://github.com/jerryc127/hexo-theme-butterfly.git butterfly

返回博客根目录，修改 _config.yml，将主题设置为：

theme: butterfly

随后安装主题相关依赖：

npm install hexo-butterfly –save
npm install hexo-renderer-pug hexo-renderer-sass –save

四、新建第一篇文章

使用 Hexo 提供的命令创建文章：

hexo new “我学会了如何搭建个人博客”

Hexo 会在以下路径生成 Markdown 文件：

D:\BlogProjects\source_posts\我学会了如何搭建个人博客.md

五、编辑文章内容

打开生成的 Markdown 文件，开始编写内容，主要包括：

  1. 安装 Hexo 的过程
  2. 安装并配置 Butterfly 主题
  3. 本地预览博客的方法
  4. 写文章与发布文章的流程

六、本地生成与预览

执行以下命令：

hexo clean
hexo generate
hexo server

随后在浏览器中访问：

http://localhost:4000/

即可在首页看到刚创建的文章。

通过以上步骤，我完成了个人博客的基本搭建，并能够在本地正常访问和预览。

搭建个人博客过程中遇到的挫折整理

一、Hexo 主题和插件加载失败

在执行 hexo clean、hexo generate 或 hexo new 等命令时，控制台反复提示错误：

ERROR Plugin load failed: hexo-butterfly
Error: EISDIR: illegal operation on a directory, read

这说明 Butterfly 相关插件加载失败。由于错误信息并不直观，初期很难判断问题出在配置、插件版本还是目录结构上，导致每一步操作都伴随报错。

二、跨盘符操作不熟悉

最初 Hexo 安装在 C 盘，但博客项目放在 D 盘。由于不熟悉 Windows 命令行的盘符切换机制，没有先输入 D:，导致多次在错误路径下执行命令，出现找不到文件或配置的问题。

三、版本兼容问题

在安装 Node.js、Hexo 以及各类 Hexo 插件时，不同版本之间存在兼容性问题。有些插件在当前 Node 版本下无法正常工作，报错信息又较为模糊，只能通过反复尝试和查阅资料解决。

四、文章显示为“无标题”

创建第一篇文章后，网页中却显示“无标题”，即使在 hexo new 命令中已经指定了标题。最终发现是 Markdown 文件的 front-matter 格式存在问题，或者主题未正确读取标题字段。

五、默认用户名和头像需要手动修改

Hexo 默认作者名为 John Doe，头像也是默认图片。如果不主动修改 _config.yml，会一直显示默认信息，不符合个人博客的使用需求。

六、本地正常但 GitHub Pages 显示异常

在本地使用 hexo server 预览时，博客样式和功能完全正常。但部署到 GitHub Pages 后，页面样式丢失，CSS 和 JS 无法加载，所有超链接点击后均显示 404。

问题的根本原因是未配置 root 参数。Hexo 默认以 / 作为根路径，而 GitHub Pages 的项目页实际路径为：

https://username.github.io/仓库名/

路径不匹配直接导致资源加载失败。

七、GitHub Pages 初次部署配置复杂

首次部署到 GitHub Pages 时，需要额外安装 hexo-deployer-git 并配置仓库地址和分支。部署过程中出现如下报错：

Author identity unknown
fatal: unable to auto-detect email address

必须手动设置 Git 的用户名和邮箱：

git config –global user.name “Your Name”
git config –global user.email “you@example.com“

否则无法完成推送。

八、部署成功但页面没有任何美化

即使 GitHub Pages 显示部署成功，页面依然是纯文本样式。这一问题与 root 配置和缓存有关，需要修改配置后执行：

hexo clean
hexo generate
hexo deploy

多次生成与部署后，样式才最终生效。

九、License 选择的困惑

在创建 GitHub 仓库时，需要选择是否添加 License，但对于 MIT、CC 系列许可证的区别并不清楚。虽然可以先随意选择一个，但仍然需要后续补充相关知识。

十、LF 与 CRLF 的警告信息

在部署过程中，Git 控制台反复提示：

warning: LF will be replaced by CRLF

虽然不影响实际使用，但在初期会误以为是严重错误，需要额外了解 Git 对不同操作系统换行符的处理机制。

心得体会

这次个人博客的搭建过程，让我切身感受到了技术实践中不可避免的挫折与反复。从环境配置到插件安装，再到主题适配和路径调整，每一个环节都可能出现意料之外的问题。尤其是 Hexo 与 Butterfly 主题的兼容性，以及本地环境与 GitHub Pages 部署环境之间的差异，让我不得不多次查阅文档并反复调试配置文件。

在这个过程中，我逐渐熟悉了 Hexo 的整体工作流程，也掌握了 Git 的基本使用方法，包括本地配置、仓库推送以及 Pages 部署。这不仅让我理解了博客系统的运行机制，也让我对前端资源加载和路径配置有了更直观的认识。

最重要的是，这次实践让我意识到，技术问题本身并不可怕，真正的难点在于是否能够保持耐心并持续尝试。每一次报错的解决，都是一次能力的积累。通过这次博客搭建，我不仅完成了一个可以长期维护和更新的个人网站，也为之后的学习与创作打下了坚实的基础。
